# コミットスコープガイド

この文書は、Conventional Commits の `scope` を一貫して選ぶための基準をまとめる。

コミットメッセージの基本形式と `type` は [COMMIT_MESSAGE_GUIDE.md](./COMMIT_MESSAGE_GUIDE.md) を参照する。

## 1. 基本方針

`scope` は、変更の主対象となる領域を短く表す。変更内容の説明ではなく、変更が属する場所または責務を示す。

- `scope` は小文字の英単語またはハイフン区切りの英単語で書く
- 既存のスコープがある場合は、同じ対象に別名を作らない
- 変更対象が明確になる場合に付け、全体に及ぶ変更では省略してよい
- 原則として 1 コミットにつき 1 つのスコープを付ける
- ディレクトリ名ではなく責務を優先する。ただし、責務が定まらない場合はディレクトリ名を使う

例:

```text
fix(auth): reject expired refresh tokens
docs(git): コミットスコープ基準の追加
chore: リポジトリ全体の改行コード統一
```

## 2. 選び方

次の優先順位で、最も具体的で安定した対象を選ぶ。

1. 機能・ドメイン: `auth`、`billing`、`search`
2. コンポーネント・モジュール: `api`、`worker`、`cli`
3. 層・基盤: `db`、`config`、`deps`
4. ドキュメント領域: `git`、`security`、`readme`

たとえば、認証 API のトークン検証を修正する場合は `api` ではなく `auth` を使う。API 全体に共通するエラー形式を変更する場合は `api` を使う。

## 3. 基本スコープ

プロジェクト固有の語彙が定まるまで、次のスコープを基本セットとして使う。

| スコープ | 対象 |
| --- | --- |
| `auth` | 認証、認可、セッション、権限 |
| `api` | API の公開仕様、エンドポイント、リクエスト・レスポンス |
| `ui` | 画面、UI コンポーネント、表示・操作 |
| `db` | スキーマ、マイグレーション、永続化 |
| `worker` | 非同期ジョブ、キュー、バッチ処理 |
| `cli` | コマンドラインインターフェース |
| `config` | アプリケーション設定、環境変数、設定ファイル |
| `deps` | 依存パッケージとロックファイル |
| `build` | ビルド定義、パッケージング、成果物生成 |
| `ci` | CI/CD のワークフロー、パイプライン |
| `test` | テスト共通基盤、フィクスチャ、テスト設定 |
| `docs` | 複数領域にまたがる一般ドキュメント |
| `readme` | README と導入手順 |

このリポジトリでは、領域別ドキュメントの変更には `git`、`python`、`security` も使う。

## 4. 判断例

| 変更内容 | 推奨スコープ | 例 |
| --- | --- | --- |
| ログイン失敗時のロック処理 | `auth` | `fix(auth): prevent repeated login attempts` |
| ユーザー検索エンドポイントの追加 | `api` | `feat(api): add user search endpoint` |
| データベースのインデックス追加 | `db` | `perf(db): add index for user search` |
| CI の依存キャッシュ設定 | `ci` | `ci: cache dependency downloads` |
| ライブラリの更新 | `deps` | `chore(deps): update pytest to 8.4.0` |
| Git ガイドの更新 | `git` | `docs(git): コミットスコープ基準の追加` |
| Python ガイドの更新 | `python` | `docs(python): Ruff設定例の更新` |
| セキュリティガイドの更新 | `security` | `docs(security): 公開サーバー対策の追記` |

## 5. 複数領域にまたがる変更

異なるスコープの変更がある場合は、まずコミットを分けられないか検討する。

- 機能実装に必要なテストは、実装と同じスコープにする
- API と UI を独立して取り込める場合は、`api` と `ui` に分ける
- 1 つの変更として分けられない場合は、主な変更対象をスコープにする
- どの対象も主ではない全体変更は、スコープを省略する

避ける例:

```text
feat(api,ui): add user search
fix(misc): 修正
chore(update): update files
```

複数スコープの表記方法はツールやホスティングサービスによって解釈が異なるため、標準運用では使わない。`misc`、`common`、`update` のように対象が分からない語も使わない。

## 6. スコープの追加・変更

新しいスコープは、次の条件を満たす場合に追加する。

- 今後も繰り返し変更される独立した領域である
- 既存スコープでは対象を適切に表せない
- チーム内で意味を共有でき、既存名と重複しない

一度使い始めたスコープの名前は、履歴検索の一貫性を保つため、安易に変更しない。
